# ForgeFlow Contract Map

ForgeFlow keeps behavior stable by treating a small set of files and commands as explicit contracts. This map records the source of truth, regeneration path, validation command, and consumers for each major surface.

## Contract surfaces

| Surface | Source of truth / owner | Regeneration command | Validation command | Known consumers |
| --- | --- | --- | --- | --- |
| `schemas/*.schema.json` | Canonical artifact contracts for task artifacts such as briefs, plans, run state, review reports, and eval records. `forgeflow_runtime/schema_versions.py` owns the current supported artifact version and future migration hook seam. | Hand-maintained; schema changes must be paired with runtime/tests. | `python3 scripts/validate_sample_artifacts.py` and `python -m pytest tests/test_validate_sample_artifacts.py -q` | Runtime artifact readers, sample fixtures, review/eval tooling, downstream agents that read ForgeFlow artifacts. |
| `schemas/policy/*.schema.json` | Canonical policy-file contracts for `policy/canonical/*.yaml`. | Hand-maintained with matching policy changes. | `python3 scripts/validate_policy.py` and `python -m pytest tests/test_validate_policy.py -q` | Policy validator, runtime policy loading, evolution policy checks, maintainers editing canonical policy. |
| `policy/canonical/*.yaml` | Human-readable canonical workflow, stage, gate, review, routing, and evolution policy. | Hand-maintained. | `python3 scripts/validate_policy.py` | Runtime orchestration, adapter prompt generation, docs, policy/evolution tests. |
| `schemas/evolution-rule.schema.json` and `examples/evolution/*` | Deterministic HARD-rule example contract and sample rules for project-local evolution rule lifecycle. | Hand-maintained examples; project adoption copies examples into `.forgeflow/evolution/rules/`. | `python3 scripts/validate_policy.py` and `python -m pytest tests/evolution -q` | Evolution inspect/list/dry-run/adopt/execute flows, project-local rule registries, policy hardening tests. |
| `forgeflow_runtime/*` | Runtime behavior implementation. Scripts should call into this layer rather than duplicate behavior. Runtime seam boundaries currently include `forgeflow_runtime/gate_evaluation.py`, `forgeflow_runtime/resume_validation.py`, `forgeflow_runtime/artifact_validation.py`, `forgeflow_runtime/task_identity.py`, `forgeflow_runtime/plan_ledger.py`, `forgeflow_runtime/stage_transition.py`, and `forgeflow_runtime/route_execution.py`. Evolution runtime seam boundaries currently include `forgeflow_runtime/evolution_rules.py`, `forgeflow_runtime/evolution_lifecycle.py`, `forgeflow_runtime/evolution_execution.py`, `forgeflow_runtime/evolution_doctor.py`, `forgeflow_runtime/evolution_promotion_plans.py`, `forgeflow_runtime/evolution_proposals.py`, `forgeflow_runtime/evolution_promotion_gates.py`, and `forgeflow_runtime/evolution_promotions.py`; approved command execution lives with rule execution rather than a separate tiny command seam. | Hand-maintained Python; extract one cohesive seam at a time from `forgeflow_runtime/orchestrator.py` or the evolution façade. | `python -m pytest tests/runtime -q`, `python -m pytest tests/evolution -q`, plus focused seam tests. | CLI scripts, Make targets, tests, future agent adapters. |
| `scripts/*.py` | Thin command-line and validation entrypoints. Script/runtime boundary: CLI parsing, printing, and exit-code handling live here; reusable behavior lives in `forgeflow_runtime/`. High-risk CLI wrappers such as `scripts/run_orchestrator.py` and `scripts/forgeflow_evolution.py` must delegate reusable orchestration/evolution behavior to runtime modules. | Hand-maintained. | `make validate` and script-specific tests under `tests/`. | Local operators, CI, README examples, plugin smoke checks. |
| `adapters/targets/*` | Handwritten adapter source and manifests for each supported agent surface. | Hand-maintained adapter target files. | `python3 scripts/validate_generated.py` and `python -m pytest tests/test_generate_adapters.py tests/test_validate_generated.py -q` | Adapter generator, plugin packages, manual install docs. |
| `adapters/generated/*` | Generated adapter output. Do not hand-edit as a source of truth. | `python3 scripts/generate_adapters.py` | `python3 scripts/validate_generated.py` | Claude/Codex/Cursor users, manual adapter installs, plugin packaging. |
| `.claude-plugin/*` and `.codex-plugin/*` | Plugin packaging manifests and packaged command surfaces. | Hand-maintained packaging metadata plus generated adapter content where referenced. | `python -m pytest tests/test_plugin_manifests.py -q` and `make validate` | Claude Code plugin install/update, Codex plugin consumers, release checks. |
| `Makefile` targets | Repo-managed execution paths for setup, validation, generation, smoke tests, and read-only local summaries. | Hand-maintained. | `make validate` | CI, README examples, scheduled maintenance, local maintainers. |
| `examples/runtime-fixtures/*` | Golden runtime examples and negative fixtures for orchestration behavior. | Hand-maintained fixtures; update with matching schema/runtime changes. | `python -m pytest tests/runtime -q` and `python3 scripts/validate_sample_artifacts.py` | Runtime tests, orchestrator smoke paths, documentation examples. |
| `docs/*.md` | Human-readable contracts and operating guidance. `docs/contract-map.md` is the index for source-of-truth boundaries. | Hand-maintained; generated outputs should point back to their source. | `python3 scripts/validate_structure.py` plus docs assertions in tests where behavior-sensitive. | Maintainers, adapter authors, users, future AI agents working in the repo. |

## Maintenance rules

- Prefer changing the canonical source first, then regenerate or validate dependent surfaces.
- Treat `adapters/targets/*` and `adapters/generated/*` as the generated adapter source/generated boundary: update adapter sources, run `python3 scripts/generate_adapters.py`, then verify with `python3 scripts/validate_generated.py`.
- Do not patch generated adapter output directly; update adapter sources and run the generator.
- Keep scripts thin: CLI parsing and printing live in `scripts/`, reusable behavior lives in `forgeflow_runtime/`.
- Add a focused validation or test assertion when a new contract surface becomes required.
- Contract map validation checks named runtime seam files exist, so seam docs cannot silently point at missing modules.
- Contract map validation checks named validation command paths exist, so command examples cannot silently rot into missing scripts or test paths.
- If a README example names a command, prefer a Make target or repo-managed script path that CI can exercise.
